# Independent Root Cause Investigation

## Investigate Failure 1: Phase4 hash mismatch (statistical_plan.yaml)
- **What hash is currently expected**: The test `test_verified_registry_hashes_match_checkout_attributes` dynamically calculates `aad0afdd8f395ecdd4853e737782c487e3b3207730e3496c9e7b4d1ddb558a07` based on simulating a `.gitattributes` `crlf` checkout, while the registry holds `96161dd17f00bd76534f3f70842775e303e926319ff59406c4ef06108df404ec`.
- **Who expects it**: `phase5/tests/test_domain_registry.py`.
- **Where the expected hash originates**: The `aad0...` hash is generated by the test reading `git check-attr eol` (which returns `crlf` on Windows) and mutating the original blob by replacing `\n` with `\r\n`.
- **What the actual file hash is**: The actual blob hash inside Git (and the bytes currently on disk) is `96161dd17f00bd76534f3f70842775e303e926319ff59406c4ef06108df404ec`, which perfectly matches the registry.
- **Whether the registry is outdated**: No. The registry accurately reflects the true `LF` blob bytes.
- **Whether the file changed after freeze**: No.
- **Whether the file belongs to**: Original freeze.
- **Classification**: OUTDATED EXPECTATION (OUTDATED TEST).

## Investigate Failure 2: Phase 4 GO report hash mismatch
- **What hash is currently expected**: `97927a12b707d65985c3db66890dd1c8be28d94009b5469f8a93379878dd729a`.
- **Who expects it**: `phase5/tests/test_domain_registry.py` (`test_registry_loads_and_resolves_verified_entries`).
- **Where the expected hash originates**: A hardcoded string inside the unit test source code.
- **What the actual file hash is**: `5766e232af92a5d6b545f8e2f61c2fe499d9da71b7760e3f9a9fab6b8054c3e1`.
- **Whether the registry is outdated**: No. The registry correctly holds the updated `5766e2...` hash.
- **Whether report changed intentionally**: Yes, as part of the authorized Phase 5 remediation/re-evaluation, but the test was forgotten and left hardcoding the legacy hash.
- **Classification**: OUTDATED TEST.

## Investigate Failure 3: Materializer schema failure
- **Current frozen row schema**: Defined in `phase4/configs/phase5_schema_freeze.json`.
- **Expected schema**: `trial_id`, `model_id`, `density`, `payload_id`, `payload_condition`, `defense_condition`, `status`.
- **Who defines expected schema**: Phase 4 scientific freeze (`phase5_schema_freeze.json`).
- **Whether task_id/task_hash were intentionally added**: No. They do not exist in the canonical `trial_order_core.csv`. They only exist as mock data inside `phase5/tests/fixtures/p11/adversarial_row.json`.
- **Whether tests were updated**: No. The fixture was left stale with legacy mock fields.
- **Whether materializer should accept them**: No. The materializer strictly and correctly enforces `FROZEN_TRIAL_FIELDS` via fail-closed behavior.
- **Whether validator is stale**: No, the materializer is correct.
- **Whether frozen artifacts are wrong**: No.
- **Classification**: STALE TEST (OUTDATED TEST).

## Investigate Failure 4: Gate0 dirty working tree failure
- **Determine whether this is simply caused by**: Uncommitted remediation work. The `git status` output confirms untracked Phase 5 audit reports (`independent_final_kaggle_readiness_audit.md`, `.json`, `.csv`) and modified validation reports.
- **Classification**: EXPECTED DEVELOPMENT STATE.

## Cross-check
There are no overlapping or incompatible versions. The registry accurately models the real artifacts on disk. The only failures originate from test fixtures and test assertions that either hardcode old hashes, apply incorrect Windows CRLF transformations, or inject legacy schema fields.

## Final Verdict

1. **Does anything require returning to Phase 1?**
No. There is no evidence of scientific inconsistencies, payload drift, or invalid schema data in the production artifacts.

2. **Does anything require regenerating Phase4?**
No. The Phase 4 frozen artifacts on disk are byte-for-byte correct and their exact blob hashes properly match the Phase 5 upstream registry. 

3. **Does anything require regenerating Phase4.5?**
No. 

4. **Does anything require only updating tests/registry?**
Yes. The three failing `pytest` assertions strictly require updating the outdated test files to align with the correctly repaired architecture and registry.

5. **Is the previous audit's recommendation: RETURN TO PHASE 1/4 REMEDIATION actually justified?**
No. The previous auditor incorrectly conflated failing test fixtures and dirty Git state (uncommitted audit reports) with cryptographic corruption. By failing to independently verify the origin of the hash expectations (which were hardcoded in tests or mutated by Windows CRLF logic), the previous audit falsely declared a scientific baseline breach. The Phase 4 baseline is intact.
